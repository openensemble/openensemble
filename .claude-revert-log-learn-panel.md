## State files this work can WRITE TO (data-loss surface)

- `users/<id>/proposals.json` — already exists; new `snoozed` status
  values + `wakeAt` field on proposals. Old code path: snoozed records
  would show up as unrecognized status; boot-load would NOT reap them
  back to pending (just keep them as a non-rendering status). Safe.
- `users/<id>/role-rules/<roleId>.md` — revoke deletes a line. To
  recover: `git`-untracked, no built-in versioning. **Mitigation:** the
  revoke endpoint writes the removed line to
  `users/<id>/role-rules/<roleId>.deleted.log` (append-only, plain text,
  with timestamp). On revert, this log file can be safely deleted.
- `users/<id>/ha-aliases.json` — revoke calls existing `deleteAlias()`.
  No built-in versioning. **Mitigation:** revoke endpoint writes a JSONL
  log to `users/<id>/ha-aliases.deleted.log` with the previous mapping
  + timestamp.
- `users/<id>/routines.json` — revoke calls `saveRoutines()` with the
  filtered list. **Mitigation:** revoke endpoint writes a JSONL log to
  `users/<id>/routines.deleted.log` with the removed routine record.

## Safety check before restart

- `npm run typecheck`
- `node --check` on every modified .mjs
- Manually fetch GET /api/learnings as the test user → expect 200
- Open the Learn drawer in the browser → expect render with at least
  the empty-state for each section
- Verify existing proposal accept/dismiss still works (don't break
  inline chat bubbles)

## Restart cost

Server restart wipes in-memory state (pairing codes, chat streams,
pending node commands, watcher in-flight ticks). The existing
`bootLoadProposals()` rehydrates proposals from disk so accepting an
inline chat bubble survives restart. New snooze status persists to
disk and rehydrates the same way.

### Architecture note
Lazy post-snapshot — no scheduler entry. On every GET /api/learnings, any
outcome record whose `checkAt` (acceptedAt + 7d) has passed and whose
postCount is still null computes its post count from proposals.json and
persists. Means a server-down-for-9d window still produces correct outcomes
when next opened.

### State files

- `scripts/smoke-learn-phases.mjs` — Phase 10 assertions

### State files
- `users/<id>/skill-overrides.json` — `{[skillId]: {disabled?, hiddenTools?: []}}`

### Behavior
- Users can disable any non-always_on skill; tools won't appear in their agents' catalogs or dispatch.
- Users can hide individual tools from any skill (no always_on guard — per-tool granularity is safe).
- Always_on skills can't be fully disabled (safety net for essential skills like coordinator, self-mgmt). Their individual tools can still be hidden.
- Overrides apply at READ time — no boot mutation, no restart needed.
